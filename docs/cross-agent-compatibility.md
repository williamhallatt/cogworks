# Cross-Agent Compatibility Matrix

**Document Date:** 2026-03-11 | **Agent Coverage:** Claude Code, GitHub Copilot, Codex/GPT-5 CLI, Generic MCP

---

## Overview

This document maps cogworks skills and generated skills across agent platforms, documenting supported invocation syntax, argument interpolation behavior, and tool restrictions. It exists because cogworks generates skills for distribution via `npx skills add`, where those skills will run on agents other than Claude Code. The document serves as the canonical reference for what works where, what's partially supported, and what requires live testing to confirm.

**Why this matters:** A skill generated by cogworks on Claude Code may use `$ARGUMENTS` interpolation or `allowed-tools` restrictions. If those features don't work on GitHub Copilot (the primary distribution channel), users will experience silent failures — arguments are lost, or tool restrictions are ignored. This matrix prevents that.

## $ARGUMENTS Interpolation

The cogworks skills include documentation and examples of `$ARGUMENTS`, `$ARGUMENTS[N]`, and `$N` placeholders. Here's where they work:

### Skill: cogworks

- **Frontmatter:** No explicit `allowed-tools` or `$ARGUMENTS` usage in cogworks/SKILL.md
- **Content references:** Mentions argument parsing in Step 1 (Gather Sources) for custom destination ("to {destination}"), but this is handled by natural language parsing, not `$ARGUMENTS` token substitution
- **Status:** No direct interpolation dependencies detected

### Skill: cogworks-encode

- **Frontmatter:** No `$ARGUMENTS` placeholders
- **Content references:** Documents the 8-phase synthesis process but does not use `$ARGUMENTS` for parameterized steps
- **Status:** No direct interpolation dependencies detected

### Skill: cogworks-learn

- **Frontmatter:** No `$ARGUMENTS` placeholders in SKILL.md frontmatter
- **Content references:** 
  - **reference.md:** Documents `$ARGUMENTS`, `$ARGUMENTS[N]`, `$N` as skill-writing patterns (lines 80-85, 254-259, 285-290)
  - **patterns.md:** Example shows "Research $ARGUMENTS thoroughly" for a template skill (`Bash(pattern-research)`), not for cogworks-learn itself
  - **examples.md:** Shows generated skill examples using `$ARGUMENTS` (e.g., "Fix GitHub issue $ARGUMENTS following our coding standards") — these are templates for *generated skills*, not cogworks-learn usage
- **Status:** cogworks-learn *documents* `$ARGUMENTS` for skill authors but does not *use* it itself

### Interpolation Support by Agent

| Placeholder | Claude Code | GitHub Copilot | Codex/GPT | MCP | Notes |
|-------------|-------------|-----------------|-----------|-----|-------|
| `$ARGUMENTS` | ✅ Confirmed | ❓ **Undefined** | ✅ Confirmed | ❓ Untested | Copilot support marked as "undefined" in risk analysis (D6). |
| `$ARGUMENTS[N]` | ✅ Confirmed | ❓ **Undefined** | ✅ Confirmed | ❓ Untested | Array indexing; support follows parent feature. |
| `$N` (shorthand) | ✅ Confirmed | ❓ **Undefined** | ✅ Confirmed | ❓ Untested | Shorthand for `$ARGUMENTS[N]`; follows same support. |
| Fallback (if no `$` support) | Uses natural language | Uses natural language | Uses natural language | Uses natural language | If `$ARGUMENTS` not recognized, skill text is passed as-is; agent interprets via language understanding. |

**Critical Gap:** GitHub Copilot's support for `$ARGUMENTS` interpolation is not documented in the risk analysis. Skills generated for Copilot that rely on `$ARGUMENTS` may silently fail. Users on Copilot will need to provide arguments in natural language (e.g., "cogworks encode these three sources as my-skill") rather than via positional tokens.

## allowed-tools Compatibility

The `allowed-tools` field in SKILL.md frontmatter restricts which tools the agent can use when the skill is active.

### cogworks Skills Usage

- **cogworks/SKILL.md:** No `allowed-tools` declared
- **cogworks-encode/SKILL.md:** No `allowed-tools` declared
- **cogworks-learn/SKILL.md:** No `allowed-tools` declared

All three core skills rely on the agent's full tool access (file reading, web fetching, bash execution, etc.). No explicit restrictions are configured.

### Tool Restriction Support by Agent

| Agent | allowed-tools Respected | Notes |
|-------|-------------------------|-------|
| Claude Code | ✅ Yes | Native support; enforced at runtime. |
| GitHub Copilot | ❓ **Undefined** | Risk analysis (D6) does not document support. If declared in a generated skill, may be ignored silently. |
| Codex/GPT-5 | ✅ Yes | Inherits OpenAI API tool restriction semantics. |
| MCP | ❓ Untested | MCP spec includes tool availability contracts; agent implementation determines enforcement. |

**Risk:** If a generated skill declares `allowed-tools: [Read, Grep]` to restrict Copilot from destructive operations, and Copilot ignores this field, the safety boundary fails silently. Users testing on Claude Code (where the field works) may not discover this until running on Copilot.

## Compatibility Note for Generated Skills

Generated skills created by cogworks are portable across Agent Skills–compatible agents but have important caveats. **Argument interpolation (`$ARGUMENTS`, `$N`)** is supported on Claude Code and Codex/GPT-5 but may not work on GitHub Copilot — if your arguments are not interpolated, provide them in natural language instead. **Tool restrictions (`allowed-tools`)** are enforced on Claude Code and Codex but enforcement on Copilot is undefined; if you intend to restrict tool access, test on your target agent and do not rely on this field for safety boundaries on Copilot without independent verification. **Reference material** in generated skills (patterns.md, reference.md, examples.md) is portable and works identically across all agents. When running a generated skill on an agent other than the one you generated it on, test core workflows first, particularly any that rely on `$ARGUMENTS` or enforce tool restrictions.

## Recommendations for Skill Authors

### For cogworks-generated Skills

1. **Document agent-specific behavior:** If a generated skill uses `$ARGUMENTS`, add a note to SKILL.md: "On agents lacking `$ARGUMENTS` support (including GitHub Copilot), provide arguments in natural language: 'cogworks encode these sources as my-skill'."

2. **Avoid exclusive reliance on `allowed-tools` for safety:** If tool restriction is critical, implement verification logic in the skill content as well. Do not rely on the field alone to prevent unwanted operations on agents where enforcement is uncertain.

3. **Test on target agents:** Before distributing a generated skill, test it on at least Claude Code and GitHub Copilot (the most common distribution targets).

### For cogworks Workflow (Orchestrator)

1. **Add generated-skill template note:** When cogworks-learn generates a skill, include a "Compatibility" section (2–3 sentences) in generated SKILL.md for agents lacking `$ARGUMENTS` support.

2. **Surface agent-specific examples:** In generated reference.md or examples.md, include examples showing both `$ARGUMENTS` syntax and natural language fallback for maximum compatibility.
