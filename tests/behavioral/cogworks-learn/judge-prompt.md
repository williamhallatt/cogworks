# LLM-as-Judge Prompt: cogworks-learn

## Purpose
Measures whether a cogworks-learn output correctly authors skills — specifically: instructing parallel tool use for independent operations, delegating high-volume tasks to subagents, accurately labeling Claude Code-specific features, correctly describing allowed-tools support breadth, including a compatibility field when CC-specific features are used, and routing requests to the appropriate mechanism when a skill is not the right artifact. This is external ground truth; a judge verdict is never stored as future training data.

## Usage
- **Model**: Must differ from the model that generated the skill being evaluated. If generated by Claude Sonnet, judge with GPT-4.1 or Gemini Pro. Never judge with the same model family.
- **Input**: The full SKILL.md content generated by cogworks-learn
- **Output format**: Structured JSON matching the schema below

---

## Judge Prompt

```
System: You are an expert evaluator assessing whether a SKILL.md produced by cogworks-learn is correctly authored for cross-agent compatibility. You are not the model that produced the skill. Your job is to surface authoring errors the generating model is most likely to overlook.

You will be given:
1. The original skill generation request (the user prompt that triggered cogworks-learn)
2. The generated SKILL.md content

Evaluate on six dimensions, then produce a JSON verdict.

---

DIMENSION 1 — PARALLEL TOOL INSTRUCTION (parallel_instruction)

Definition: Skills that require multiple independent file reads or tool calls MUST instruct the agent to make those calls in parallel, not sequentially. A skill for "analyzing TypeScript files across a repo" must not instruct the agent to read files one at a time.

Fail signals:
- Skill instructs sequential reads ("First read file A, then read file B")
- Skill uses numbered steps for operations that have no ordering dependency
- Skill does not mention parallel execution despite multiple independent data-gathering operations

Pass signals:
- Skill explicitly states "make all independent tool calls in parallel"
- Parallel reads are grouped in the same instruction step
- Skill distinguishes between operations that must be sequential and those that can be parallelized

EXAMPLE (from test case cogworks-learn-parallel-001):
  Request: Skill for analyzing code quality across multiple TypeScript files.
  PASS output includes: "Make all independent tool calls in parallel — read all target files in a single parallel batch before analysis."
  FAIL output: "Step 1: Read file 1. Step 2: Read file 2. Step 3: Analyze both files."

---

DIMENSION 2 — SUBAGENT DELEGATION (subagent_delegation)

Definition: Skills that involve high-volume result tasks (test suite runs, batch log analysis, large-scale file processing) MUST include guidance to delegate to a subagent to avoid context exhaustion. Keeping verbose output in the main context is an authoring defect.

Fail signals:
- Skill instructs agent to run test suite and return full output to main context
- No mention of subagent delegation for operations that produce large outputs
- Skill suggests summarizing in-context instead of delegating

Pass signals:
- Skill explicitly states "Delegate this task to a subagent"
- Subagent receives the high-volume operation; only summary returns to main context
- For Claude Code-specific batch operations, context:fork is mentioned and labeled CC-only

EXAMPLE (from test cases cogworks-learn-subagent-001 and cogworks-learn-subagent-002):
  Request: Skill for running test suites and summarizing failures.
  PASS output includes: "Delegate this task to a subagent. The subagent runs the full test suite and returns only the failure summary to the main context."
  FAIL output: "Run the test suite with npm test and capture all output for analysis."

  Request: Skill for batch log analysis (Claude Code).
  PASS output includes: "Use context: fork [Claude Code only] to process logs in an isolated subagent."
  FAIL output: "Analyze all log files and return findings." (no delegation, no CC label)

---

DIMENSION 3 — CLAUDE CODE FEATURE LABELING (cc_feature_labeling)

Definition: Features that are Claude Code-specific — $ARGUMENTS, disable-model-invocation, context:fork — MUST be labeled as "[Claude Code only]" wherever they appear. Presenting them as universal skill features is a compatibility defect that breaks skill portability.

Fail signals:
- $ARGUMENTS used without "[Claude Code only]" label
- disable-model-invocation described without noting it is CC-specific
- context:fork presented as a general skill feature
- Any of these described as "standard", "universal", or part of the agentskills.io spec

Pass signals:
- Each CC-specific feature has "[Claude Code only]" annotation inline or in a dedicated note
- Feature is explained with context of what supporting agents can expect instead
- skill notes that agentskills.io spec does not include $ARGUMENTS

EXAMPLE (from test cases cogworks-learn-arguments-001 and cogworks-learn-subagent-002):
  Request: Skill using $ARGUMENTS for a file path parameter.
  PASS output includes: "$ARGUMENTS [Claude Code only] — passes the user-provided argument string directly. Not in the agentskills.io spec; other agents parse arguments from the user message."
  FAIL output: "The skill accepts a file path via $ARGUMENTS." (no CC label)

---

DIMENSION 4 — ALLOWED-TOOLS ACCURACY (allowed_tools_accuracy)

Definition: The allowed-tools frontmatter field restricts which tools an agent may use during skill execution. This field is broadly supported across agent implementations (16 of 18 agents). It must not be described as experimental, Claude Code-only, or narrowly supported.

Fail signals:
- allowed-tools described as "experimental"
- allowed-tools described as a Claude Code-only feature
- allowed-tools described as "not widely supported" or "limited support"
- Support breadth omitted entirely when the feature is discussed

Pass signals:
- allowed-tools described as "broadly supported" with approximate support count (16/18 or equivalent)
- Skill notes any known exceptions or agents that do not support it
- Description does not overstate or understate adoption

EXAMPLE (from test case cogworks-learn-allowed-tools-001):
  Request: Skill with restricted tool access using allowed-tools.
  PASS output includes: "allowed-tools is broadly supported across 16/18 agents. List only the tools this skill requires."
  FAIL output: "Note: allowed-tools is a Claude Code-specific feature and may not be available in other agents."

---

DIMENSION 5 — COMPATIBILITY FIELD (compatibility_field)

Definition: Any skill that uses Claude Code-specific features ($ARGUMENTS, disable-model-invocation, context:fork) MUST include a `compatibility:` field in its YAML frontmatter that identifies which features are CC-specific. Omitting this field when CC features are present is a portability defect.

Fail signals:
- Skill uses $ARGUMENTS but has no compatibility: field
- Skill uses disable-model-invocation but has no compatibility: field
- Skill uses context:fork but has no compatibility: field
- compatibility: field is present but does not list the CC-specific features used

Pass signals:
- compatibility: field lists each CC-specific feature used in the skill
- Field format is valid YAML within the frontmatter block
- Field is present in frontmatter, not just mentioned in body prose

EXAMPLE (from test case cogworks-learn-compatibility-001):
  Request: Skill using disable-model-invocation and $ARGUMENTS.
  PASS frontmatter includes:
    ```yaml
    compatibility:
      claude-code: [disable-model-invocation, $ARGUMENTS]
    ```
  FAIL: Skill uses $ARGUMENTS throughout but has no compatibility: field in frontmatter.

---

DIMENSION 6 — SCOPE APPROPRIATENESS (scope_appropriateness)

Definition: When a request describes a use case better served by persistent config, hooks, or subagent definitions, cogworks-learn MUST identify this and recommend the appropriate mechanism rather than (or in addition to) generating a skill. Producing a skill without acknowledging the better-fit mechanism is a routing defect.

Fail signals:
- Skill generated for always-on rules (style, conventions) without recommending persistent config (CLAUDE.md, copilot-instructions.md, AGENTS.md)
- Skill generated for deterministic enforcement without recommending hooks
- Skill generated for task orchestration with restricted tools/model without recommending subagent definitions
- No "Why a skill?" note when the use case is borderline

Pass signals:
- Explicit redirect to the appropriate mechanism with rationale
- "Why a skill?" note present when a skill is generated for a borderline case
- null when the request is clearly a normal skill use case (task-specific workflow, domain knowledge)

EXAMPLE (from test cases cogworks-learn-persistent-001, cogworks-learn-hook-001, cogworks-learn-subagent-def-001):
  Request: Skill for enforcing code style rules that should apply to every session.
  PASS output includes: "This is better served by persistent configuration (CLAUDE.md, copilot-instructions.md) — always-on rules should not be packaged as a skill."
  FAIL output: Generates a complete SKILL.md for style enforcement without mentioning persistent config.

  Request: Skill that blocks git push without running tests first.
  PASS output includes: "This is a deterministic enforcement case — use a hook (PreToolUse lifecycle event, exit code 2 to block) rather than a skill."
  FAIL output: Generates a SKILL.md that instructs the agent to check for tests before pushing.

  Request: Skill for a specialized code review agent with restricted tools and a custom model.
  PASS output includes: "This describes a subagent definition (custom agent config), not a skill. Define it as a subagent with restricted tools and model selection."
  FAIL output: Generates a SKILL.md for code review without mentioning subagent definitions.

---

SCORING INSTRUCTIONS

For each dimension, assign a score from 0.0 to 1.0:
  1.0 = criterion fully met, no issues
  0.5–0.9 = partially met, some issues present but not disqualifying
  0.1–0.4 = substantially violated, criterion mostly fails
  0.0 = criterion completely absent or inverted

Note: Dimensions only apply when relevant to the skill being generated. If the skill has no independent parallel operations, set parallel_instruction to null and exclude it from the verdict calculation. Apply the same logic to other dimensions. For scope_appropriateness, set to null when the request is clearly a normal skill use case — only evaluate when the request describes a use case that could be better served by persistent config, hooks, or subagent definitions.

Overall verdict:
  "pass" — all applicable dimensions ≥ 0.7
  "fail" — any applicable dimension < 0.5
  "uncertain" — borderline cases where evidence is ambiguous

Your reasoning must cite specific text from the SKILL.md content. Quote the passages you are judging.
```

---

## Output Schema

```json
{
  "verdict": "pass | fail | uncertain",
  "confidence": "float 0.0–1.0 — how confident the judge is in the verdict",
  "dimension_scores": {
    "parallel_instruction": "float 0.0–1.0 | null (null if dimension not applicable)",
    "subagent_delegation": "float 0.0–1.0 | null",
    "cc_feature_labeling": "float 0.0–1.0 | null",
    "allowed_tools_accuracy": "float 0.0–1.0 | null",
    "compatibility_field": "float 0.0–1.0 | null",
    "scope_appropriateness": "float 0.0–1.0 | null"
  },
  "reasoning": "string — specific quoted evidence from the SKILL.md supporting the verdict",
  "applicable_dimensions": ["list of dimension keys that were evaluated (non-null)"]
}
```

All fields are required. `reasoning` must reference text from the actual SKILL.md, not the request. `null` dimensions must be listed in `applicable_dimensions` exclusions — do not include them in verdict calculation.

## Calibration Notes

These reference the `notes` and `expected_content` fields from quality test cases:

**cogworks-learn-parallel-001** (parallel tool use):
Skills for multi-file operations that omit parallel instruction are defective by design. The judge should not award partial credit for sequential steps that happen to produce the same result — the instruction pattern matters because it affects agent efficiency and context consumption.

**cogworks-learn-subagent-001** and **cogworks-learn-subagent-002** (subagent delegation):
A skill that runs a test suite and collects all output in-context will exhaust context on large repos. The judge must treat in-context collection of high-volume results as a **fail**, not a stylistic choice.

**cogworks-learn-arguments-001** (CC feature labeling):
The expected_content for this case explicitly requires: `["Claude Code", "$ARGUMENTS", "compatibility", "not in agentskills.io spec"]` and forbids `["universal", "standard"]`. The judge must check that all four expected signals are present.

**cogworks-learn-compatibility-001** (compatibility field):
The compatibility field must appear in **frontmatter YAML**, not just be mentioned in body prose. A skill that discusses CC features in its instructions but lacks the frontmatter field fails this criterion.

**cogworks-learn-allowed-tools-001** (allowed-tools accuracy):
Expected: `["allowed-tools", "broadly supported", "16/18"]`. Forbidden: `["experimental", "Claude Code-only"]`. The judge must verify the support breadth claim is present and that no restrictive language is used.

**cogworks-learn-hook-001** (scope appropriateness — hook routing):
Expected: `["hook", "deterministic"]`. The request describes deterministic enforcement (block git push without tests) which is better served by a hook than a skill. The judge must verify the output recommends a hook and explains why deterministic enforcement should not rely on AI interpretation.

**cogworks-learn-subagent-def-001** (scope appropriateness — subagent definition routing):
Expected: `["subagent", "definition", "tools", "model"]`. The request describes a custom agent config with restricted tools and model selection — a subagent definition, not a skill. The judge must verify the output recommends a subagent definition rather than generating a SKILL.md.

**cogworks-learn-subagent-type-001** (subagent delegation — type selection):
Expected: `["Explore", "read-only", "subagent", "context: fork"]`. A batch log analysis skill should select the Explore subagent type (read-only, Haiku) since log analysis requires only reading. The judge must verify correct type selection rationale and CC-specific labeling.

**cogworks-learn-subagent-dispatch-001** (subagent delegation — parallel dispatch):
Expected: `["parallel", "subagent", "background", "concurrent"]`. Three independent research tasks should use background dispatch for concurrent execution. The judge must verify the output instructs parallel/background subagent dispatch, not sequential foreground calls.

## Cross-Model Independence Note

If the SKILL.md was generated by **Claude Sonnet or Claude Opus**: judge with **GPT-4.1**, **GPT-5-mini**, or **Gemini Pro**.
If generated by a **GPT model**: judge with **Claude Haiku** or **Gemini Pro**.
If generated by **Gemini**: judge with any Claude or GPT model.

Never use the same model family as the generator. A model that authored a skill with unlabeled CC features is likely to also evaluate that skill as portable — this constraint prevents that rationalization.
